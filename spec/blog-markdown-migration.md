# 博客 Markdown 迁移需求文档

## 需求描述

当前博客内容存储在 [`src/data/blog-posts.ts`](/home/hfy/code/personal-site/src/data/blog-posts.ts) 中，采用 TypeScript 常量硬编码的方式维护文章标题、摘要、导语和正文段落。该方式适合少量静态文章，但不适合后续批量接入已有的图文 Markdown 博客内容。

后续计划将大量现成的 Markdown 图文博客迁移到个人网站中，并支持长期持续新增文章。迁移后的博客需要能够较低成本地维护正文、封面图、配图、摘要、日期、标签等内容，并继续兼顾 SEO、静态生成、仓库内管理和部署稳定性。

本需求聚焦于将博客内容源从“代码内硬编码”迁移为“文件型内容源”，优先支持 Markdown 图文文章的站内展示，不引入后台编辑系统，不将博客正文存入数据库。

---

## 需求目标

1. **降低内容维护成本**：新增或修改博客时，以编辑 Markdown 文件为主，不再手动维护 TypeScript 文章数据结构。
2. **支持图文内容迁移**：支持将现有 Markdown 文章及配套图片较完整地迁移到站内，并保留清晰的资源组织方式。
3. **保持现有路由与 SEO 能力**：博客列表页、详情页、静态路由生成、sitemap 等能力继续保留，并基于新内容源工作。
4. **保证长期可扩展性**：为后续标签、目录、相关文章、RSS、阅读时长、代码高亮等博客能力预留空间。
5. **控制系统复杂度**：优先采用文件系统 + 构建期静态生成方案，不在当前阶段引入数据库博客表、CMS 后台或在线编辑能力。

---

## 技术方案

### 方案选择

推荐采用“**Markdown/MDX + frontmatter + 本地内容目录 + Next.js 静态生成**”的方案，其中首选 Markdown，只有在明确需要在文章中嵌入 React 组件时再升级为 MDX。

不推荐当前阶段采用 Prisma 或其他数据库存储博客正文，原因如下：

- 现有文章源本身即为 Markdown 文件，直接落文件系统迁移成本最低。
- 博客更新频率相对可控，不需要为“内容存储”额外引入后台管理复杂度。
- 文件型内容源更适合图文资源和文章正文一起管理，也更适合 Git 版本控制。

### 内容目录结构

建议新增 `content/blog/` 作为博客内容根目录，每篇文章使用独立目录管理正文与图片资源，例如：

```text
content/
  blog/
    2025-12-06-ai-coding-experience/
      index.md
      cover.png
      img-01.png
      img-02.png
    2026-01-15-example-post/
      index.md
      cover.jpg
```

每篇文章的 `index.md` 使用 frontmatter 维护元信息，例如：

```md
---
title: AI 编程经验分享
slug: ai-coding-experience
date: 2025-12-06
excerpt: 从 Copilot 到 Cursor、Claude Code、Codex 等 Agent 的实践体会
tags: [AI, 编程, Agent]
cover: ./cover.png
published: true
---

正文内容
```

### 渲染与读取方式

- 博客内容从 `content/blog/` 目录扫描读取。
- 在 `src/service/` 或 `src/lib/` 中新增博客内容服务层，负责：
  - 扫描文章目录
  - 解析 frontmatter
  - 读取 Markdown 正文
  - 按 `slug` 获取文章
  - 生成博客列表数据
  - 处理文章内相对图片路径
- `/blog` 列表页与 `/blog/[slug]` 详情页继续保留现有路由结构，但数据来源从 `src/data/blog-posts.ts` 切换为内容目录。
- `generateStaticParams`、metadata、sitemap 等静态能力继续由博客内容服务层提供支持。

### 图片资源策略

图文博客迁移的重点不只是正文，还包括图片资源组织方式。建议遵循以下规则：

- 每篇文章的配图和封面与文章正文放在同一目录下。
- Markdown 正文内的图片优先使用相对路径引用，如 `![说明](./img-01.png)`。
- 封面图通过 frontmatter 单独声明，不依赖从正文中推断。
- 避免将所有博客图片平铺到 `public/` 目录，降低命名冲突和维护成本。

### 分阶段落地

建议按以下阶段推进：

1. **第一阶段：内容源切换**
   - 建立 `content/blog/` 目录规范
   - 实现 Markdown 文章扫描、读取、列表与详情渲染
   - 迁移 1 到 2 篇现有文章验证链路
2. **第二阶段：图文能力完善**
   - 支持文章目录内图片
   - 统一文章排版样式
   - 支持标签、封面、摘要、发布日期等元数据展示
3. **第三阶段：博客体验增强**
   - 增加目录、上一篇/下一篇、阅读时长、RSS、代码高亮等附加能力

## 待讨论事项

1. **Markdown 还是 MDX**
   - 默认建议先使用 Markdown。
   - 如后续明确需要在文章中嵌入 React 组件、交互 Demo 或定制卡片，再评估升级为 MDX。

2. **历史文章 slug 规则**
   - 需要确认 slug 是否完全手动维护，还是可由目录名或文件名派生。
   - 一旦对外发布，slug 应尽量保持稳定，避免旧链接失效。

3. **旧文章图片清洗范围**
   - 需要确认历史 Markdown 中图片是本地相对路径、外链图床，还是混合形式。
   - 若存在大量外链图，是否需要同步下载并本地化存储，需要进一步评估工作量。

4. **现有静态文章如何过渡**
   - 需要确认当前 [`src/data/blog-posts.ts`](/home/hfy/code/personal-site/src/data/blog-posts.ts) 中的文章是直接迁移为首篇 Markdown 样板，还是暂时保留兼容一段时间。

5. **是否需要草稿机制**
   - 当前建议通过 frontmatter 中的 `published` 字段控制文章是否出现在列表页和详情页路由中。
   - 若后续有更复杂的审核、定时发布需求，再单独扩展。
